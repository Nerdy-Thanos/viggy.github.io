about

Data through, platforms under

I'm a data platform engineer with six-plus years building high-throughput batch and streaming pipelines and the self-service cloud platforms teams run them on — including three-plus years inside tier-1 UK banks (Citi and Barclays), where reliability isn't negotiable. The systems I've shipped process millions of records a day and hold 99.9% uptime.

I own systems end to end: architecture, infrastructure-as-code, deployment, and production support. Much of my work is multiplied through other teams — Terraform modules, Helm charts, and CI/CD pipelines built once and adopted across the organisation as the paved road.

Right now I'm at Citi, enabling Databricks and large-scale Spark for tenant teams on EKS and GKE. Before that: Barclays (streaming, serverless ETL, dbt on Fargate), a fintech startup (Airflow, Kafka, Snowflake), and a consumer-goods manufacturer (computer vision, Spark). I hold an MSc in Data Science from the University of Glasgow, along with AWS and Databricks certifications.

Bank-wide pub-sub notification service

A standard publish-and-consume interface for applications across Barclays.

Barclays · 2024–26

Problem. Application teams across the bank were each wiring their own point-to-point notification integrations — duplicated effort, inconsistent reliability, no shared contract.

Built. An end-to-end pub-sub notification service with a single standard interface for publishing and consuming events, with near real-time workflows orchestrated through AWS Step Functions and DynamoDB.

Impact. Adopted as the standard interface for applications across the bank, running at 2–5 minute end-to-end latency with 99.9% uptime.

  • Step Functions
  • DynamoDB
  • Lambda
  • S3

Ephemeral CI/CD test clusters

Throwaway vCluster environments that cut test pipelines from 35 to 10 minutes.

Citi · 2026

Problem. Automated API tests ran against shared, long-lived clusters — slow to provision, contended between teams, and a 35-minute drag on every pipeline run.

Built. Orchestration for ephemeral vCluster virtual clusters on EKS and GKE, spun up per pipeline run and torn down after, giving every test job an isolated Kubernetes environment.

Impact. Test pipeline runtimes dropped from 35 to 10 minutes — roughly 70% faster feedback for every team using the pipeline.

  • vCluster
  • EKS
  • GKE
  • Tekton

dbt transformation layer on Fargate

40+ dbt models turning raw nested JSON into query-optimised Iceberg tables.

Barclays · 2024–26

Problem. ~400,000 nested-JSON API records arriving daily were unusable for analytics in their raw form, and transformation logic was scattered and hard to test.

Built. An ingestion pipeline on AWS Lambda, S3 and Glue feeding a transformation layer of dbt-core running on AWS Fargate — versioned, tested SQL models materialising into Apache Iceberg tables.

Impact. 40+ models delivering clean, query-optimised data from ~400k records a day, with transformations managed as reviewable code rather than ad-hoc scripts.

  • dbt-core
  • Fargate
  • Iceberg
  • Glue
  • Lambda

Distributed serverless ETL

Step Functions pipeline processing 2M+ rows in four to five minutes.

Barclays · 2023–24

Problem. Large batch loads needed to land in Aurora PostgreSQL quickly and predictably, without a standing cluster to babysit or pay for.

Built. A distributed serverless ETL pipeline on AWS Step Functions with fan-out processing, plus reusable CloudFormation templates so other data domains could deploy the same stack without resource conflicts.

Impact. 2M+ rows processed in 4–5 minutes; shared templates cut deployment time by 40% and CI/CD automation cut manual integration effort by 75%.

  • Step Functions
  • Aurora PostgreSQL
  • CloudFormation
  • Jenkins
